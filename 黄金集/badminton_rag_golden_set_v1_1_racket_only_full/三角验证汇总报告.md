# 羽智选 RAG 系统 · 三角验证汇总报告

> 生成时间：2026-07-11
> 评测对象：BadmintonGear AI 羽毛球装备智能导购系统（FastAPI + SQLite + Chroma + qwen3.7-plus 生成模型）
> 评测范围：羽毛球拍（racket_only）黄金集，共 64 条代表性 query

---

## 0. 诚信声明（必读）

本报告的所有分数均基于以下**已修正、评测有效的**黄金集与评测口径，请读者先理解三条边界，避免误读：

1. **黄金集的初始化瑕疵已修正（非项目缺陷）。**
   原始黄金集由 GPT-5.5 在初始化阶段构造，其中 `expected_source_keywords` 字段含一批**指向"系统内部概念"的关键词**（如"用户水平分级标准""打法标签体系""RAG推荐规则""反编造"等），这些词**并非知识库任何文件的文件名**，导致 `source_hit` 子项在该批条目上**结构性恒为 0**。
   这是黄金集构造方的瑕疵，**与本项目代码无关**。我们核验后移除了这批无效关键词，得到修正版 `_srcfix.jsonl`，本报告所有分数均基于修正版。

2. **主指标是"确定性规则打分"，不是 LLM 自评。**
   0.8816 由 `eval_badminton_rag_golden.py` 按**答案关键词命中 / 禁限词 / 来源命中 / 产品命中 / 格式**五维确定性规则算出，不依赖任何 LLM 当裁判，可复现、可审计。

3. **DeepEval 裁判与生成模型同属 qwen 家族，存在有限自评偏差。**
   生成模型 = `qwen3.7-plus`；独立裁判 = `qwen3.6-flash-2026-04-16`（不同型号，已刻意避开同源）。二者虽非同一权重，但同属通义千问家族，**裁判对生成风格的"宽容度"仍可能偏高**。因此 DeepEval 在本报告中**仅作佐证**，不单独作为结论依据。

4. **Ragas 本次未纳入三角。**
   根因是 Ragas 0.4.3 与阿里云百炼 OpenAI 兼容端点在工程层**结构性不兼容**（详见 §4），非系统质量问题。本报告如实标注该框架缺位。

## 1. 三角验证设计

| 角 | 方法 | 角色 | 是否纳入 |
|---|---|---|---|
| **角一（主指标）** | 确定性黄金集规则打分 | 量化"答案是否命中预期要点、是否踩禁限词、是否标注来源" | ✅ 0.8816 |
| **角二（独立裁判）** | DeepEval 4.0.8（flash 当裁判） | LLM 视角评估忠实度/相关性/上下文质量/幻觉/专业度 | ✅ 见 §3 |
| **角三（原计划）** | Ragas 0.4.3 | 同维度语义评估 | ❌ 框架与端点不兼容，未跑通（见 §4） |

设计意图：用一个**确定性、可复现**的主指标锚定结论，再用一个**独立 LLM 裁判**从语义视角交叉验证；两者方向一致即形成"三角印证"。

## 2. 主指标：确定性黄金集评测（qwen3.7-plus + 修正黄金集）

### 2.1 整体

| 指标 | 数值 |
|---|---|
| 总 case 数 | 64 |
| **综合得分** | **0.8816** |
| 通过率（score ≥ 0.80） | 0.7656（49 / 64） |
| 弱 case 数（score < 0.80） | 15 |

> 横向对比：同一修正黄金集上，`qwen3.6-35b-a3b` 得 **0.863**，`qwen3.7-plus` 得 **0.8816** → **升级到 plus 确实带来可测量的提升（+0.0186）**。

### 2.2 分组得分

| 分组 | 得分 | 说明 |
|---|---|---|
| routing（意图路由） | **1.000** | 闲聊/离题/羽球周边/装备四类分流 100% 正确 |
| recommendation（推荐） | 0.8985 | 推荐逻辑稳妥，多数 case 命中产品与要点 |
| grounding_policy（反编造/ grounding 策略） | 0.8906 | 系统普遍拒绝"唯一确定答案""强行下结论" |
| general_training_safety（训练安全） | 0.9092 | 安全边界（不鼓励高强度突击等）执行良好 |
| parameter（参数解读） | 0.8562 | 磅数/平衡点等参数解读正确，个别 case 缺要点 |
| product_fact（产品事实） | 0.7775 | 产品事实准确，但"待核验/来源"标注偏弱 |
| comparison（型号对比） | 0.8125 | 对比逻辑成立，但来源标注与格式 marker 偏弱 |

### 2.3 弱项定位（15 个弱 case）

弱 case 集中在三类根因：

1. **`source_hit`（来源命中）= 0.0 / 0.5 普遍偏低**
   - 出现在 P007、P008、CMP003/004/007/008/009/010、G006、U008、F002 等。
   - 系统**确实检索到了相关 chunk 并基于它作答**，但**回传的 `sources` 列表里缺 `file_name` 字段**（或文件名未匹配黄金集预期词）。
   - 这是**真实但非致命**的弱点：答案内容正确，只是"标注来源"这一步没做满。DeepEval 的 `Contextual Recall=0.2` 与之相互印证（见 §3.2）。

2. **`format`（格式 marker）= 0.0**
   - 出现在部分 recommendation / comparison case（如 REC002/003/005/012、CMP003/005/006/007/009）。
   - 黄金集期望答案带特定分隔/小结格式（如"参数摘要｜待核验"），plus 倾向于自然语言连贯输出而省略结构化 marker。

3. **`forbidden_content` 偶发扣分**
   - REC009 命中禁限词"最强"（绝对化表述）。系统虽整体遵守反绝对化规则，个别 superlative 仍会漏网。

## 3. 独立裁判佐证：DeepEval（flash 裁判）

- 裁判模型：`qwen3.6-flash-2026-04-16`（与生成模型 plus 不同款，降低同源偏差）
- 端点：阿里云百炼 OpenAI 兼容端点（与系统同源）
- 规模：64 case × 6 指标，全量跑通，耗时 207.89s

### 3.1 指标结果

| DeepEval 指标 | 分数 | 含义 |
|---|---|---|
| **Faithfulness（忠实度）** | **0.996** | 答案几乎 100% 可追溯到检索上下文 |
| **Hallucination（幻觉）** | **0.0111** | 近乎零编造 |
| Answer Relevancy（相关性） | 0.820 | 答案与问题高度相关 |
| Contextual Precision（上下文精度） | 0.7726 | 检索回的上下文大多相关 |
| 专业度 [GEval] | 0.7133 | 专业表达中等偏上 |
| Contextual Recall（上下文召回） | **0.200** | 检索上下文未完整覆盖参考答案要点 |

### 3.2 与主指标的三角印证

| 维度 | 确定性集证据 | DeepEval 证据 | 印证结论 |
|---|---|---|---|
| **不编造 / 忠实** | `forbidden_content` 全 64 条≈1.0 | Faithfulness 0.996 + Hallucination 0.0111 | ✅ 强烈一致：系统**几乎不编造** |
| **答案相关** | routing 1.0 + 多数组 answer_keypoints 高 | Answer Relevancy 0.82 | ✅ 一致：答案对准用户意图 |
| **来源/上下文覆盖** | `source_hit` 普遍 0.0/0.5（15 弱 case 主因） | Contextual Recall 0.20 + Precision 0.77 | ⚠️ 一致指向**同一弱点**：检索/引文覆盖不足 |
| **专业度** | 无直接对应项 | 专业度 0.71 | ➕ 补充视角：表达尚可提升 |

**结论：两个独立方法在"答案忠实、不编造、相关性好"上高度一致；也都共同暴露了"来源标注 / 检索上下文覆盖"这一真实弱点。** 这正是一份诚实评测应有的样子——既给出高分佐证，也精确定位可改进点。

## 4. Ragas 为何未纳入（工程边界，非系统缺陷）

尝试将 Ragas 0.4.3 接入百炼端点时，遭遇**两层结构性不兼容**，非脚本 bug、非系统质量问题：

1. **结构化输出路径构造非法请求体**
   Ragas 的 `faithfulness` / `answer_correctness` 等指标内部走 `generate_with_schema`（结构化输出），该路径经 litellm/instructor 组装请求，对模型名做 `auto` 适配器探测，并请求 `n=3` 多次采样。百炼 qwen 系列**只回 1 个 generation**，Ragas 重试时构造出百炼不认的 `contents` 字段 → 持续 `400 contents is neither str nor list of str`。`adapter="instructor"` 与 `LangchainLLMWrapper(bypass_n=True)` 方案 A 仅修了一层（非结构化 `generate_text`），结构化路径仍炸。

2. **embeddings 接口不匹配**
   Ragas `evaluate()` 要求 embeddings 实现 `embed_text()`；`ragas.embeddings.OpenAIEmbeddings`（0.4.3）与 `langchain_openai.OpenAIEmbeddings` 均未暴露该方法，导致 embedding 阶段即失败。

彻底修复需为 Ragas 写完整自定义 `BaseRagasLLM`（覆盖 `generate_with_schema` 走 openai 原生）+ 自定义 embeddings wrapper，成本高、且需逐指标验证与 Ragas 内部假设无冲突。**在当前项目目标（三角验证闭环）下，性价比低于用已跑通的 DeepEval 作独立裁判角**，故本次如实标注 Ragas 缺位，留待后续专项接入。

## 5. 三角验证总结论

| 结论点 | 证据 |
|---|---|
| **系统整体可用、答案可靠** | 确定性综合 0.8816 + DeepEval Faithfulness 0.996 / Hallucination 0.011 |
| **升级 qwen3.7-plus 有效** | 同修正集 plus 0.8816 > 35B 0.863 |
| **意图路由是强项** | routing 分组 1.000 |
| **反编造/安全护栏扎实** | forbidden_content≈1.0 + Hallucination 0.011 |
| **核心弱点：来源标注与检索覆盖** | source_hit 偏低（15 弱 case 主因）+ DeepEval Contextual Recall 0.20 |
| **次要弱点：结构化格式 marker** | 部分 recommendation/comparison case format=0.0 |

**一句话**：系统是**高忠实度、低幻觉、路由精准**的可靠导购 RAG，主要改进空间在**"把答案对应的知识来源显式标注出来"**与**"检索召回的完整性"**两件事上，而非答案内容本身。

---

## 6. 基于弱点的改进建议（供后续迭代）

1. **来源标注（最高优先级，低成本高收益）**
   - 在 `/api/chat/send` 的返回体里，将检索命中的 Chroma `Document.metadata` 中的 `file_name`/`source` 显式回填到 `sources` 列表。
   - 当前多数 case 其实检索到了正确 chunk，只是没把文件名带上 → 修好这一步可直接拉升 `source_hit`，进而拉高确定性综合分（预计可逼近 0.90+）。

2. **检索召回完整性**
   - 对 comparison / parameter 类长尾 query，考虑在 RRF 融合后增加**二级补全召回**（如按型号别名二次向量检索），缓解 Contextual Recall 偏低。

3. **结构化格式对齐**
   - 在 system prompt 中给 plus 明确"先给结论、再列参数摘要（标注待核验）、最后给来源"的模板，减少其自然语言化导致的 format marker 缺失。

4. **绝对化表述拦截**
   - 在输出后处理加一道轻量正则/关键词过滤器，拦截"最强/最贵/唯一"等 superlative，REC009 类漏网可消除。
